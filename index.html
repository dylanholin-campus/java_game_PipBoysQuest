Diagramme de classes UML - Pip-Boy's Quest

Mise à jour selon les classes Java actuelles, avec relations structurelles explicites

Relations UML principales

Dépendance
Main --> GameWindow / Menu / DatabaseManager / Game
`Main` initialise l'interface, la persistance et la session de jeu.
Composition
Game ◆── Board / Dice
La session de jeu possède son plateau et son dé.
Association
Game ── Menu / Character / DatabaseManager
`Game` orchestre les menus, le héros courant et la persistance.
Composition
Board ◆── Cell[*]
Le plateau contient une collection de cases spécialisées.
Héritage
EmptyCell / EnemyCell / ItemsCell --|> Cell
Trois variantes concrètes de case spécialisent le contrat `Cell`.
Héritage
Warrior / Wizard --|> Character
Les deux classes jouables partagent l'état commun du personnage abstrait.
Association
EnemyCell ── Enemy
Une case hostile embarque un ennemi concret pour déclencher un combat.
Association
ItemsCell ── OffensiveEquipment / DefensiveEquipment
Une case de loot transporte soit une arme, soit un équipement défensif.
Association
Character ── OffensiveEquipment / DefensiveEquipment
Le héros conserve son arme et son équipement défensif courants.
Héritage
Weapon --|> OffensiveEquipment
Toutes les armes du jeu sont construites a partir de `Weapon`.
Héritage
Shield / Potion --|> DefensiveEquipment
Les équipements défensifs sont spécialisés entre bouclier et potion.
Héritage
StandardPotion / LargePotion --|> Potion
Les presets de potions dérivent de la classe `Potion`.
Héritage
Dragon / Sorcerer / Goblin --|> Enemy
Les ennemis concrets partagent la base abstraite `Enemy`.
Dépendance
Board / DatabaseManager --> WeaponCatalog / PotionCatalog
Les catalogues centralisent les données d'armes et de potions (noms, valeurs, quantités).
Héritage
OutOfBoardException --|> Exception
Exception métier dédiée au dépassement de la dernière case.
Légende
--|> héritage   ◆── composition   ── association   --> dépendance
Ce bloc complète les cartes de classes ci-dessous pour montrer explicitement les relations UML.
